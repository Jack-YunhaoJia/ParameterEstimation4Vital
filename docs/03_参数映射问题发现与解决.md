# Vital VST3 参数映射：问题发现与解决全记录

> 本文档记录了在构建 Vital 合成器数据生产流水线过程中，发现并解决参数映射问题的完整思维链条。
> 适用于论文撰写中的数据集制作方法论部分。

## 1. 背景与问题起源

### 1.1 初始方案

我们的数据生产流水线需要将随机采样的 45 维参数向量加载到 Vital VST3 插件中，渲染出对应的音频。初始方案使用 pedalboard 库加载 Vital VST3，通过 `plugin.raw_state` 接口将 Vital 的 JSON 预设直接写入插件。

### 1.2 第一个问题：预设参数未生效

**现象**：生产流水线成功渲染了 100,000+ 音频样本，但人工抽听发现所有样本听起来几乎完全相同。

**诊断过程**：
1. 对比不同预设的渲染音频波形，发现波形几乎一致
2. 检查 `plugin.raw_state` 写入后的插件状态，发现参数值未改变
3. 查阅 pedalboard 源码和 VST3 规范

**根因**：Vital 的 VST3 接口不支持通过 `raw_state`（二进制状态块）加载 JSON 格式的预设。`raw_state` 期望的是 Vital 内部的二进制序列化格式，而非 JSON。写入 JSON 字节后，插件静默忽略了无效数据，保持默认状态。

**解决方案**：改用 `plugin.parameters[name].raw_value` 逐参数设置。pedalboard 暴露了 Vital 的所有 VST3 参数，每个参数的 `raw_value` 是 [0, 1] 归一化值。

## 2. 参数名映射问题

### 2.1 命名差异发现

**现象**：使用 `plugin.parameters[vital_name].raw_value` 设置参数时，大量参数报 KeyError。

**诊断过程**：
1. 枚举 pedalboard 暴露的所有参数名，与 Vital JSON 参数名对比
2. 发现系统性的命名差异

**发现的映射规则**：

| Vital JSON 命名 | pedalboard 命名 | 示例 |
|-----------------|----------------|------|
| `osc_X_*` | `oscillator_X_*` | `osc_1_level` → `oscillator_1_level` |
| `env_X_*` | `envelope_X_*` | `env_1_attack` → `envelope_1_attack` |
| `*_on` | `*_switch` | `chorus_on` → `chorus_switch` |
| `*_dry_wet` | `*_mix` | `chorus_dry_wet` → `chorus_mix` |
| `*_keytrack` | `*_key_track` | `filter_1_keytrack` → `filter_1_key_track` |

此外还有若干不规则映射（如 `osc_1_random_phase` → `oscillator_1_phase_randomization`），通过特殊映射表处理。

**解决方案**：实现 `_vital_name_to_pedalboard()` 函数，包含规则映射和特殊映射表。

## 3. 参数值映射：核心难题

### 3.1 问题发现

**现象**：参数名映射解决后，渲染的音频仍然存在严重问题：
- 不同八度（C3/C4/C5）的音频听起来完全相同
- 约 26.7% 的样本是静音的

**关键洞察**：pedalboard 的 `raw_value` 始终是 [0, 1] 归一化值，但 Vital JSON 中的参数值是物理值（如 `env_1_attack=2.0` 表示 2 秒，`filter_1_cutoff=60` 表示 MIDI note 60）。直接将物理值写入 `raw_value` 会导致严重错误。

### 3.2 三类映射关系的发现

通过系统性实验（对比 Vital 默认预设的 JSON 值与 pedalboard 默认 raw_value），我们发现 Vital 的 672 个参数与 pedalboard raw_value 之间存在三种不同的映射关系：

**类型 1：直接映射（304 个参数）**
- Vital JSON 值 == raw_value
- 特征：参数范围本身就是 [0, 1]
- 示例：`osc_1_level=0.707` → `raw=0.707`

**类型 2：线性归一化（180 个参数）**
- `raw = (vital_value - pb_min) / (pb_max - pb_min)`
- 特征：参数有明确的物理范围，pedalboard 报告了 min/max
- 示例：`osc_1_transpose=0`（范围 [-48, 48]）→ `raw = (0+48)/(48+48) = 0.5`

**类型 3：非线性映射（188 个参数）**
- 复杂的内部映射函数，无法通过简单公式转换
- 策略：对于非 CORE_PARAMS 的非线性参数，跳过设置，保持插件默认值

### 3.3 分类方法

我们通过以下实验方法对每个参数进行分类：

```python
# 对每个参数，比较 Vital 默认 JSON 值经过不同映射后是否匹配 pedalboard 默认 raw_value
for param_name in all_params:
    vital_default = vital_json[param_name]
    pb_raw_default = plugin.parameters[pb_name].raw_value
    pb_min = plugin.parameters[pb_name].min_value
    pb_max = plugin.parameters[pb_name].max_value

    # 测试直接映射
    if abs(vital_default - pb_raw_default) < 1e-4:
        classify_as("direct")

    # 测试线性归一化
    normalized = (vital_default - pb_min) / (pb_max - pb_min)
    if abs(normalized - pb_raw_default) < 1e-4:
        classify_as("normalize")

    # 否则为非线性
    classify_as("nonlinear")
```

## 4. 包络参数的 Power-Law 映射

### 4.1 问题发现

**现象**：修复了基本的值映射后，静音率仍然高达 26.7%。分析发现大量样本的 `env_1_attack` 被设置为极长的值。

**诊断过程**：

1. 检查一个 `env_1_attack=2.0`（2 秒）的预设：
   - 使用线性归一化：`raw = 2.0 / 4.0 = 0.5`
   - 但 pedalboard 的 `string_value` 显示 `raw=0.5` 对应 "2.00 s"... 看似正确？

2. 进一步检查 `env_1_attack=4.0`（4 秒，CORE_PARAMS 最大值）：
   - 线性归一化：`raw = 4.0 / 4.0 = 1.0`
   - pedalboard `string_value` 显示 `raw=1.0` 对应 **"32.00 s"**
   - **这意味着 raw=1.0 不是 4 秒，而是 32 秒！**

3. 系统性测量多个 raw_value 对应的实际时间：

| raw_value | 实际时间（秒） | 线性预期（秒） |
|-----------|---------------|---------------|
| 0.00 | 0.000 | 0.0 |
| 0.25 | 0.125 | 1.0 |
| 0.50 | 2.000 | 2.0 |
| 0.75 | 10.125 | 3.0 |
| 1.00 | 32.000 | 4.0 |

### 4.2 映射函数推导

观察数据模式：
- `raw=0.5 → 2.0s`：`2.0 = 32 × 0.5^4`？验证：`32 × 0.0625 = 2.0` ✓
- `raw=0.25 → 0.125s`：`32 × 0.25^4 = 32 × 0.00390625 = 0.125` ✓
- `raw=0.75 → 10.125s`：`32 × 0.75^4 = 32 × 0.31640625 = 10.125` ✓

**确认映射函数**：

```
正向: time_seconds = 32 × raw^4
逆向: raw = (time_seconds / 32) ^ 0.25
```

### 4.3 影响分析

使用错误的线性映射时：
- `env_1_attack=4.0s` → `raw=1.0` → 实际 32 秒（远超 2 秒渲染窗口，导致静音）
- `env_1_attack=2.0s` → `raw=0.5` → 实际 2.0 秒（碰巧正确，因为 `32 × 0.5^4 = 2.0`）
- `env_1_attack=3.0s` → `raw=0.75` → 实际 10.125 秒（严重偏差）

使用正确的 power-law 映射后：
- `env_1_attack=4.0s` → `raw=(4/32)^0.25 = 0.595` → 实际 4.0 秒 ✓
- `env_1_attack=3.0s` → `raw=(3/32)^0.25 = 0.554` → 实际 3.0 秒 ✓

### 4.4 验证

通过 pedalboard 的 `string_value` API 对 11 个测试值进行精确验证：

```python
test_times = [0.0, 0.001, 0.01, 0.1, 0.5, 1.0, 2.0, 3.0, 4.0, 0.25, 0.05]
for t in test_times:
    raw = (t / 32.0) ** 0.25 if t > 0 else 0.0
    param.raw_value = raw
    actual = parse_time(param.string_value)  # 从 "2.00 s" 解析
    assert abs(actual - t) < 0.001  # 全部通过 ✓
```

修复后静音率从 26.7% 降至 0%（5 预设测试）和 11.3%（50 预设测试，均为合理的低能量组合）。

## 5. 滤波器截止频率的 MIDI Note 映射

### 5.1 问题发现

**现象**：不同八度（C3/C4/C5）的渲染音频听起来相同。

**诊断**：`filter_1_cutoff` 在 Vital JSON 中的值是 MIDI note 编号（如 60 = C4），但之前被当作线性归一化参数处理。pedalboard 报告的 min/max 范围与实际映射不一致。

### 5.2 映射函数确认

通过实验确认：

```
raw = (midi_note - 8) / 128
```

验证：
- Vital 默认 `filter_1_cutoff=60`（C4）
- pedalboard 默认 `raw=0.40625`
- 计算：`(60 - 8) / 128 = 52 / 128 = 0.40625` ✓

范围：MIDI note 8 → raw=0.0，MIDI note 136 → raw=1.0。

## 6. 反馈参数的特殊处理

### 6.1 问题发现

**现象**：`chorus_feedback` 和 `delay_feedback` 的范围是 [-1, 1]，但在一次代码编辑中被意外从非线性参数列表中移除。

**影响**：这两个参数的 Vital 值 0.0（无反馈）应映射到 `raw=0.5`（范围中点），但被当作直接映射处理后变成 `raw=0.0`（最大负反馈）。

### 6.2 解决方案

将 `chorus_feedback` 和 `delay_feedback` 恢复到非线性参数列表中。由于它们是 CORE_PARAMS，会通过 CORE_PARAMS 回退路径进行线性归一化：

```
raw = (vital_value - (-1)) / (1 - (-1)) = (vital_value + 1) / 2
```

验证：`vital=0.0` → `raw=0.5` → pedalboard `string_value` = "0%" ✓

## 7. 最终映射策略总结

`_vital_value_to_raw()` 函数的完整映射优先级：

```
1. 特殊参数
   ├── volume: 线性增益 → dB → 归一化
   ├── unison_detune: sqrt 映射 raw = sqrt(val/100)
   └── unison_voices: 离散映射 raw = (val-1)/15

2. Envelope 时间参数 (env_1_attack/decay/release)
   └── Power-law: raw = (time_seconds / 32) ^ 0.25

3. Filter cutoff (filter_1_cutoff)
   └── MIDI note: raw = (midi_note - 8) / 128

4. 线性归一化参数 (180个，pattern匹配)
   └── raw = (vital_value - pb_min) / (pb_max - pb_min)

5. 直接映射参数 (304个)
   └── raw = vital_value (已在 [0,1] 范围内)

6. CORE_PARAMS 回退 (非线性但在45参数中)
   └── raw = (vital_value - core_min) / (core_max - core_min)

7. 未知参数
   └── 跳过，保持插件默认值
```

## 8. 经验教训与方法论启示

### 8.1 VST 插件参数接口的陷阱

1. **`raw_state` 不等于预设文件**：VST3 的状态序列化格式是插件特定的，不能假设与插件的原生预设格式相同
2. **`raw_value` 不等于物理值**：VST3 规范要求参数归一化到 [0, 1]，但归一化方式由插件决定
3. **参数命名不统一**：VST3 暴露的参数名可能与插件内部命名不同

### 8.2 非线性映射的普遍性

合成器参数中非线性映射非常普遍，原因是：
- **人类感知是对数的**：频率、时间、响度等参数使用对数或幂律映射，使旋钮在感知上均匀分布
- **物理约束**：包络时间使用 `time = max × raw^power` 映射，使短时间段有更高的控制精度
- **历史原因**：不同合成器厂商有各自的内部映射约定

### 8.3 验证方法

对于闭源 VST 插件的参数映射，推荐以下验证方法：

1. **`string_value` API**：pedalboard 暴露了 VST3 的 `string_value`，可以获取参数的人类可读表示（如 "2.00 s"、"440 Hz"），用于验证映射正确性
2. **默认值对比**：加载默认预设，对比 JSON 值和 raw_value，快速分类映射类型
3. **渲染验证**：设置已知参数值，渲染音频，通过听觉或频谱分析验证效果
4. **系统性扫描**：对关键参数进行 raw_value 扫描（0.0 到 1.0），记录 string_value，拟合映射函数

### 8.4 对数据集质量的影响

参数映射的正确性直接决定了训练数据的质量：
- **映射错误 → 参数-音频不对应**：模型学到的是错误的映射关系
- **映射错误 → 无效样本增多**：如包络映射错误导致大量静音样本
- **映射错误 → 音色多样性降低**：如滤波器截止频率映射错误导致不同预设听起来相似

正确的参数映射是整个数据生产流水线的基石。
